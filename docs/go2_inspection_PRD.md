# Go2 机器狗巡检系统 - 需求文档

> 本文档用于 OpenCode 新 session，指导开发完整的 Go2 巡检仿真系统。

---

## 1. 项目目标

开发一套基于 Web 的 Unitree Go2 机器狗巡检仿真系统，实现：

1. **仿真环境** - 在服务器上运行 MuJoCo 物理仿真
2. **Web 控制界面** - 从 Windows 浏览器远程操作
3. **3D 可视化** - 实时显示机器人状态和环境
4. **建图功能** - 扫描并保存区域地图
5. **地图复用** - 导入地图进行巡检任务

### 最终工作流

```
┌─────────────────────────────────────────────────────────────┐
│                      工作流程                                │
│                                                             │
│  Step 1: 建图阶段                                            │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                 │
│  │ 启动仿真 │ → │ 控制扫描 │ → │ 生成地图 │                 │
│  └─────────┘    └─────────┘    └─────────┘                 │
│                                    ↓                        │
│                              保存地图文件                    │
│                                    ↓                        │

│  ┌─────────┐    ┌─────────┐    ┌─────────┐                 │
│  │ 导入地图 │ → │ 加载仿真 │ → │ 执行巡检 │                 │
│  └─────────┘    └─────────┘    └─────────┘                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 技术栈

| 组件 | 技术选型 | 备选方案 |
|------|----------|----------|
| 物理仿真 | MuJoCo (Python绑定) | - |
| Web后端 | FastAPI + uvicorn | - |
| 前端框架 | 原生HTML/JS | Vue.js, React |
| 3D渲染 | **Three.js** | Babylon.js, PlayCanvas |
| 点云可视化 | Three.js Points | Potree.js, Potree2 |
| 2D地图显示 | Canvas API | Leaflet.js, Mapbox |
| 通信协议 | WebSocket (实时) + REST (控制) | - |

### 2.3 3D可视化方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **Three.js** ✅ | 轻量、灵活、社区活跃 | 需要自己搭建场景 | 机器人3D模型+环境 |
| Babylon.js | 功能丰富、文档好 | 包体积较大 | 复杂3D场景 |
| Potree.js | 专为点云优化 | 不适合动态物体 | 大规模点云地图 |
| PlayCanvas | 可视化编辑器 | 需要账号 | 游戏级项目 |

**推荐方案**: Three.js (主场景) + Potree.js (点云地图显示，可选)

## 4. 数据模型

### 4.1 地图文件格式

```json
{
  "version": "1.0",
  "type": "occupancy_grid",  // 或 "point_cloud"
  "created_at": "2026-03-22T20:00:00",
  "metadata": {
    "resolution": 0.05,       // 栅格大小(米)
    "width": 200,             // 栅格宽度
    "height": 200,            // 栅格高度
    "origin": {"x": 0, "y": 0, "z": 0},
    "robot_model": "go2"
  },
  "data": [...]               // 栅格值或点云数据
}
```

### 4.2 仿真状态数据

```json
{
  "time": 12.345,
  "position": {"x": 1.2, "y": 0.5, "z": 0.35},
  "orientation": {"w": 1.0, "x": 0, "y": 0, "z": 0},
  "yaw": 0.0,
  "velocity": {"x": 0.2, "y": 0.0, "z": 0.0},
  "joint_positions": [0.0, 0.67, -1.34, ...]
}
```

### 4.3 LiDAR 点云数据

```json
{
  "frame_id": 1234,
  "timestamp": 12.345,
  "robot_pose": {"x": 1.2, "y": 0.5, "yaw": 0.0},
  "points": [[x1, y1], [x2, y2], ...]  // robot frame
}
```

## 5. API 设计

### 5.1 REST API（控制命令）

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/connect` | POST | 启动仿真 |
| `/api/disconnect` | POST | 停止仿真 |
| `/api/stand_up` | POST | 站起 |
| `/api/stand_down` | POST | 趴下 |
| `/api/stop` | POST | 停止移动 |
| `/api/move` | POST | 发送速度命令 {vx, vy, yaw} |
| `/api/status` | GET | 获取当前状态 |

### 5.2 REST API（地图相关）

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/map/start_mapping` | POST | 开始建图 |
| `/api/map/stop_mapping` | POST | 停止建图 |
| `/api/map/save` | POST | 保存当前地图 {name} |
| `/api/map/list` | GET | 列出所有地图 |
| `/api/map/load/{id}` | POST | 加载地图 |
| `/api/map/delete/{id}` | DELETE | 删除地图 |
| `/api/map/export/{id}` | GET | 导出地图文件 |

### 5.3 WebSocket（实时数据）

- 路径: `/ws`
- 频率: 20Hz
- 数据: 状态 + LiDAR 点云 + 地图更新

## 8. 关键实现要点

### 8.1 仿真器要点
- MuJoCo 必须在 background thread 运行（不阻塞 Web 服务）
- 使用 threading.Lock 保护共享状态
- 控制力要足够大（fx > 500）才能驱动机器人

### 8.2 LiDAR 数据要点
- 点云必须在 **robot frame** 生成（相对于机器人朝向）
- 前端显示时转换到 world frame
- 生成角度范围：±70°（模拟 L1 LiDAR）

### 8.3 前端控制要点
- 必须使用 setInterval 持续发送命令（20Hz）
- 松开摇杆/键盘后发送零命令停止
- 右转 = 负 yaw_rate

### 8.4 建图要点
- 每帧 LiDAR 数据需要结合机器人位姿转换到 world frame
- 占据栅格使用概率更新（贝叶斯更新）
- 地图文件应包含边界信息和分辨率
